Grants and Privileges

This document explains how privileges are granted to roles on various Snowflake objects in the Terraform configuration.

Overview

Grants in Snowflake control access to objects by assigning privileges to roles. In this Terraform configuration, privilege grants are automatically managed within each module. Each module (database, schema, tables, views, warehouse) handles its own privilege grants, ensuring consistent access control.

Automatic Grant Management

Module-Based Grants

All modules that create objects automatically handle privilege grants:

  • Database Module: Grants database privileges via role_privileges in database_map
  • Schema Module: Grants schema privileges via role_privileges in schema_map
  • Tables Module: Grants table privileges via role_privileges in table_map
  • Views Module: Grants view privileges via role_privileges in view_map
  • Warehouse Module: Grants warehouse privileges via role_privileges in warehouse_map

Benefits

  • Consistency: Grants are managed alongside objects
  • Simplicity: No need for separate grant resources
  • Maintainability: Easier to understand and modify
  • Safety: Grants are always in sync with objects

Database Grants

Configuration

Database privileges are configured within the database module:

module "ALL_DATABASES" {
  source = "./modules/snowflake/database"
  
  database_map = {
    "MY_DB" = {
      comment                     = "My database"
      data_retention_time_in_days = 1
      role_privileges = [
        {
          account_role_name = module.ALL_ROLES.ROLES.ANALYST.name
          privileges        = ["USAGE"]
        },
        {
          account_role_name = module.ALL_ROLES.ROLES.DATA_ENGINEER.name
          privileges        = ["USAGE", "CREATE SCHEMA"]
        }
      ]
      with_grant_option = false
    }
  }
}

Available Database Privileges

  • USAGE: Use the database (required to access schemas/tables)
  • MODIFY: Modify database settings
  • CREATE SCHEMA: Create schemas in the database
  • MONITOR: View database usage and metrics
  • OWNERSHIP: Full ownership of the database
  • ALL PRIVILEGES: All of the above

Schema Grants

Configuration

Schema privileges are configured within the schema module:

module "ALL_SCHEMAS" {
  source = "./modules/snowflake/schema"
  
  schema_map = {
    "MY_SCHEMA" = {
      database_name       = module.ALL_DATABASES.DATABASES.MY_DB.name
      with_managed_access = false
      is_transient        = false
      role_privileges = [
        {
          account_role_name = module.ALL_ROLES.ROLES.ANALYST.name
          privileges        = ["USAGE"]
        }
      ]
    }
  }
}

Available Schema Privileges

  • USAGE: Use the schema (required to access tables/views)
  • MODIFY: Modify schema settings
  • CREATE TABLE: Create tables in the schema
  • CREATE VIEW: Create views in the schema
  • CREATE FILE FORMAT: Create file formats
  • CREATE STAGE: Create stages
  • CREATE FUNCTION: Create functions
  • CREATE PROCEDURE: Create procedures
  • CREATE SEQUENCE: Create sequences
  • CREATE TASK: Create tasks
  • CREATE PIPE: Create pipes
  • CREATE STREAM: Create streams
  • MONITOR: View schema usage and metrics
  • OWNERSHIP: Full ownership of the schema
  • ALL PRIVILEGES: All of the above

Table Grants

Configuration

Table privileges are configured within the tables module:

module "ALL_TABLES" {
  source = "./modules/snowflake/tables"
  
  table_map = {
    "MY_TABLE" = {
      database_name = module.ALL_DATABASES.DATABASES.MY_DB.name
      schema_name   = module.ALL_SCHEMAS.SCHEMAS.MY_SCHEMA.name
      comment       = "My table"
      columns = [ ... ]
      role_privileges = [
        {
          account_role_name = module.ALL_ROLES.ROLES.ANALYST.name
          privileges        = ["SELECT"]
        },
        {
          account_role_name = module.ALL_ROLES.ROLES.DATA_ENGINEER.name
          privileges        = ["SELECT", "INSERT", "UPDATE", "DELETE"]
        }
      ]
      with_grant_option = false
    }
  }
}

Available Table Privileges

  • SELECT: Query the table
  • INSERT: Insert rows into the table
  • UPDATE: Update rows in the table
  • DELETE: Delete rows from the table
  • TRUNCATE: Truncate the table
  • REFERENCES: Reference the table in foreign keys
  • OWNERSHIP: Full ownership of the table
  • ALL PRIVILEGES: All of the above

View Grants

Configuration

View privileges are configured within the views module:

module "ALL_VIEWS" {
  source = "./modules/snowflake/views"
  
  view_map = {
    "MY_VIEW" = {
      database_name = module.ALL_DATABASES.DATABASES.MY_DB.name
      schema_name   = module.ALL_SCHEMAS.SCHEMAS.MY_SCHEMA.name
      statement     = "SELECT * FROM MY_TABLE"
      comment       = "My view"
      role_privileges = [
        {
          account_role_name = module.ALL_ROLES.ROLES.ANALYST.name
          privileges        = ["SELECT"]
        }
      ]
      with_grant_option = false
    }
  }
}

Available View Privileges

  • SELECT: Query the view
  • REFERENCES: Reference the view in other objects
  • OWNERSHIP: Full ownership of the view
  • ALL PRIVILEGES: All of the above

Note: Views are read-only, so INSERT, UPDATE, DELETE privileges are not applicable.

Warehouse Grants

Configuration

Warehouse privileges are configured within the warehouse module:

module "ALL_WAREHOUSES" {
  source = "./modules/snowflake/warehouse"
  
  warehouse_map = {
    "MY_WH" = {
      warehouse_type       = "STANDARD"
      warehouse_generation = "2"
      role_privileges = [
        {
          account_role_name = module.ALL_ROLES.ROLES.ANALYST.name
          privileges        = ["USAGE"]
        }
      ]
    }
  }
}

Available Warehouse Privileges

  • USAGE: Use the warehouse for queries
  • OPERATE: Resume/suspend the warehouse
  • MODIFY: Change warehouse settings
  • MONITOR: View warehouse usage and metrics
  • ALL PRIVILEGES: All of the above

Grant Option

The with_grant_option property allows the grantee to grant the same privileges to other roles:

database_map = {
  "MY_DB" = {
    role_privileges = [ ... ]
    with_grant_option = true  # Grantees can grant to others
  }
}

Use Cases:

  • Delegating privilege management
  • Allowing role administrators to manage access
  • Flexible access control

Security Consideration:

  • Use carefully as it allows privilege escalation
  • Typically used for administrative roles
  • Default is false for security

Privilege Hierarchy

Privileges follow a hierarchy in Snowflake:

  1. Database USAGE → Required to access schemas
  2. Schema USAGE → Required to access tables/views
  3. Table/View SELECT → Required to query data

Example Complete Access Chain:

# 1. Database access
module "ALL_DATABASES" {
  database_map = {
    "MY_DB" = {
      role_privileges = [
        {
          account_role_name = module.ALL_ROLES.ROLES.ANALYST.name
          privileges        = ["USAGE"]
        }
      ]
    }
  }
}

# 2. Schema access
module "ALL_SCHEMAS" {
  schema_map = {
    "MY_SCHEMA" = {
      database_name = module.ALL_DATABASES.DATABASES.MY_DB.name
      role_privileges = [
        {
          account_role_name = module.ALL_ROLES.ROLES.ANALYST.name
          privileges        = ["USAGE"]
        }
      ]
    }
  }
}

# 3. Table access
module "ALL_TABLES" {
  table_map = {
    "MY_TABLE" = {
      database_name = module.ALL_DATABASES.DATABASES.MY_DB.name
      schema_name   = module.ALL_SCHEMAS.SCHEMAS.MY_SCHEMA.name
      role_privileges = [
        {
          account_role_name = module.ALL_ROLES.ROLES.ANALYST.name
          privileges        = ["SELECT"]
        }
      ]
    }
  }
}

Best Practices

  1. Principle of Least Privilege: Grant only necessary privileges
  2. Explicit Grants: Explicitly grant each required privilege
  3. Use USAGE First: Grant USAGE on database/schema before object privileges
  4. Avoid Grant Option: Use with_grant_option = false unless necessary
  5. Document Grants: Add comments explaining why grants are needed
  6. Role-Based: Grant to roles, not individual users
  7. Review Regularly: Regularly review and audit grants

Common Patterns

Read-Only Access

# Database: USAGE
# Schema: USAGE
# Table: SELECT
# View: SELECT

Read-Write Access

# Database: USAGE
# Schema: USAGE
# Table: SELECT, INSERT, UPDATE, DELETE

Full Access

# Database: ALL PRIVILEGES
# Schema: ALL PRIVILEGES
# Table: ALL PRIVILEGES

Troubleshooting

Role Not Found

Error:

Error: Role does not exist

Solution:

  • Ensure roles are created before objects
  • Verify role names in role_privileges match module outputs
  • Check that roles module is applied before other modules

Insufficient Privileges

Error:

Error: 001003 (42000): SQL access control error: Insufficient privileges

Solution:

  • Ensure provider uses appropriate role (SYSADMIN for objects, SECURITYADMIN for roles/users)
  • Verify user can assume required roles
  • Check that user has necessary privileges

Missing USAGE Privilege

Error:

Error: Cannot access object - missing USAGE privilege

Solution:

  • Grant USAGE on database first
  • Grant USAGE on schema before table/view privileges
  • Follow privilege hierarchy
