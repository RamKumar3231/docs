Views Module

The Views module manages Snowflake views with automatic privilege grant management.

Overview

The Views module (modules/snowflake/views) creates Snowflake views within schemas and automatically grants privileges to specified roles. This module provides a complete solution for view management including SQL statement definitions and access control.

Module Location

modules/snowflake/views/
├── main.tf        # View resource definitions and grants
├── variables.tf   # Input variables
└── outputs.tf     # Output values

Usage

Basic View

module "ALL_VIEWS" {
  source = "./modules/snowflake/views"
  
  providers = {
    snowflake = snowflake.snowflake_sysadmin
  }
  
  view_map = {
    "MY_VIEW" = {
      database_name = module.ALL_DATABASES.DATABASES.MY_DB.name
      schema_name   = module.ALL_SCHEMAS.SCHEMAS.MY_SCHEMA.name
      statement     = <<-SQL
        SELECT 
          ID,
          NAME
        FROM MY_TABLE
        WHERE ACTIVE = TRUE;
      SQL
      comment       = "My view"
      role_privileges = [
        {
          account_role_name = module.ALL_ROLES.ROLES.ANALYST.name
          privileges        = ["SELECT"]
        }
      ]
    }
  }
}

Aggregated View (Actual Example)

This matches the actual view in main.tf:

module "ALL_VIEWS" {
  source = "./modules/snowflake/views"
  
  providers = {
    snowflake = snowflake.snowflake_sysadmin
  }
  
  view_map = {
    "AGG_FACEBOOK_ADS" = {
      database_name = module.ALL_DATABASES.DATABASES.TEST_DB.name
      schema_name   = module.ALL_SCHEMAS.SCHEMAS.ADS.name
      statement     = <<-SQL
        select AD_DATE, COUNT(1) AS COUNT_ADS from ${module.ALL_TABLES.TABLES.FACEBOOK_ADS.name} GROUP BY AD_DATE;
      SQL
      comment       = "Aggregated view for facebook ads"
      role_privileges = [
        {
          account_role_name = module.ALL_ROLES.ROLES.MARKETING.name
          privileges        = ["SELECT"]
        }
      ]
      with_grant_option = false
    }
  }
}

Input Variables

view_map

Type: map(object)
Required: Yes
Description: Map of view names to view configurations

View Configuration Object

FieldTypeRequiredDefaultDescription
database_namestringYes-Name of the parent database
schema_namestringYes-Name of the parent schema
statementstringYes-SQL SELECT statement defining the view
commentstringNo""Comment describing the view
depends_onlist(string)No[]Optional list of dependent resource addresses
role_privilegeslist(object)Yes-List of role privilege grants
with_grant_optionboolNofalseWhether grants include grant option

Role Privilege Object

FieldTypeRequiredDescription
account_role_namestringYesName of the role receiving privileges
privilegeslist(string)YesList of privileges: “SELECT”, “REFERENCES”, “OWNERSHIP”, “ALL PRIVILEGES”

Resources Created

snowflake_view.VIEWS

Creates views using for_each over the view_map.

Properties:

  • name: View name (from map key)
  • database: Parent database name
  • schema: Parent schema name
  • statement: SQL SELECT statement
  • comment: View comment

snowflake_grant_privileges_to_account_role.VIEW_GRANTS

Automatically grants view privileges to roles based on the role_privileges configuration.

Properties:

  • account_role_name: Role receiving privileges
  • privileges: List of privileges
  • on_schema_object.object_type: “VIEW”
  • on_schema_object.object_name: Fully qualified view name (database.schema.view)
  • with_grant_option: Whether grant includes grant option

Dependencies:

  • Depends on snowflake_view.VIEWS to ensure views exist before granting

Outputs

VIEWS

Type: map(object)

A map of all created view resources, keyed by view name.

Example Output:

{
  "MY_VIEW" = {
    name     = "MY_VIEW"
    database = "MY_DB"
    schema   = "MY_SCHEMA"
    # ... other attributes
  }
}

Usage:

# Reference a view
view_name = module.ALL_VIEWS.VIEWS.MY_VIEW.name

SQL Statement Format

The statement property contains the SQL SELECT statement that defines the view:

Using Heredoc Syntax

statement = <<-SQL
  SELECT 
    COLUMN1,
    COLUMN2
  FROM TABLE_NAME
  WHERE CONDITION;
SQL

Benefits:

  • Multi-line SQL support
  • Better readability
  • Preserves formatting

Using String Literal

statement = "SELECT COLUMN1, COLUMN2 FROM TABLE_NAME WHERE CONDITION;"

Use when:

  • Simple single-line queries
  • No special formatting needed

Referencing Tables

You can reference tables using Terraform interpolation:

statement = <<-SQL
  SELECT * FROM ${module.ALL_TABLES.TABLES.MY_TABLE.name}
  WHERE ACTIVE = TRUE;
SQL

Dependencies

Views should explicitly declare dependencies on tables they reference using the depends_on field:

view_map = {
  "MY_VIEW" = {
    # ... other properties ...
    depends_on = [
      "module.ALL_TABLES.snowflake_table.TABLES[\"MY_TABLE\"]"
    ]
  }
}

Note: The depends_on field in the view_map is for documentation purposes. Terraform will automatically detect dependencies from the SQL statement when it references table names.

Available Privileges

  • SELECT: Query the view
  • REFERENCES: Reference the view in other objects
  • OWNERSHIP: Full ownership of the view
  • ALL PRIVILEGES: All of the above

Note: Views are read-only, so INSERT, UPDATE, DELETE privileges are not applicable.

Provider Requirements

The module requires the snowflake_sysadmin provider alias because:

  • Creating views requires SYSADMIN privileges
  • Granting view privileges requires SYSADMIN privileges

Dependencies

Internal Dependencies

  • snowflake_grant_privileges_to_account_role.VIEW_GRANTS depends on snowflake_view.VIEWS

External Dependencies

  • Database Module: Views must reference existing databases
  • Schema Module: Views must reference existing schemas
  • Tables Module (usually): Views typically reference tables
  • Roles Module: Views reference roles from the roles module for privilege grants

Examples

Simple View

module "ALL_VIEWS" {
  source = "./modules/snowflake/views"
  
  providers = {
    snowflake = snowflake.snowflake_sysadmin
  }
  
  view_map = {
    "ACTIVE_USERS" = {
      database_name = module.ALL_DATABASES.DATABASES.MY_DB.name
      schema_name   = module.ALL_SCHEMAS.SCHEMAS.PUBLIC.name
      statement     = <<-SQL
        SELECT *
        FROM USERS
        WHERE STATUS = 'ACTIVE'
          AND LAST_LOGIN >= CURRENT_DATE - 30;
      SQL
      comment       = "Currently active users"
      role_privileges = [
        {
          account_role_name = module.ALL_ROLES.ROLES.ANALYST.name
          privileges        = ["SELECT"]
        }
      ]
    }
  }
}

Aggregated View

module "ALL_VIEWS" {
  source = "./modules/snowflake/views"
  
  providers = {
    snowflake = snowflake.snowflake_sysadmin
  }
  
  view_map = {
    "DAILY_SUMMARY" = {
      database_name = module.ALL_DATABASES.DATABASES.MY_DB.name
      schema_name   = module.ALL_SCHEMAS.SCHEMAS.ANALYTICS.name

        SELECT 
          DATE_TRUNC('day', EVENT_DATE) AS DAY,
          COUNT(*) AS EVENT_COUNT,
          SUM(AMOUNT) AS TOTAL_AMOUNT
        FROM EVENTS
        GROUP BY DATE_TRUNC('day', EVENT_DATE);

      comment       = "Daily aggregated event summary"
      role_privileges = [
        {
          account_role_name = module.ALL_ROLES.ROLES.ANALYST.name

        }
      ]
    }
  }
}

Best Practices

  1. Descriptive Names: Use clear, descriptive view names
  2. Comments: Always add comments explaining view purpose
  3. SQL Formatting: Use heredoc syntax for multi-line SQL
  4. Access Control: Grant SELECT to roles that need access
  5. Minimal Privileges: Grant only necessary privileges
  6. Database/Schema Reference: Always reference databases and schemas from modules
  7. Table References: Reference tables using module outputs when possible

Troubleshooting

View Already Exists

Error:

Error: 2003: Object already exists

Solution:

  • Import existing view: terraform import module.ALL_VIEWS.snowflake_view.VIEWS["VIEW_NAME"] "DB.SCHEMA.VIEW"
  • Or remove view from Snowflake manually
  • Or use a different view name

Table Not Found

Error:

Error: Table does not exist

Solution:

  • Ensure tables are created before views
  • Verify table names in SQL statement
  • Check that tables exist in Snowflake
  • Reference tables using module outputs: module.ALL_TABLES.TABLES.TABLE_NAME.name

Invalid SQL Statement

Error:

Error: SQL compilation error

Solution:

  • Verify SQL syntax is correct
  • Test SQL statement directly in Snowflake
  • Check that referenced objects exist
  • Review Snowflake SQL documentation

Insufficient Privileges

Error:

Error: 001003 (42000): SQL access control error: Insufficient privileges

Solution:

  • Ensure provider uses SYSADMIN role
  • Verify user can assume SYSADMIN role
  • Check that user has SELECT on underlying tables
  • Verify user has necessary privileges
