Snowflake Terraform Infrastructure Documentation

Project Structure

This document explains the structure and organization of the Terraform project.

Directory Layout

terraform/
├── main.tf                    # Main configuration with all modules
├── provider.tf                # Provider configuration
├── variables.tf               # Variable definitions
├── outputs.tf                 # Output definitions
├── terraform.tfvars           # Variable values (not in version control)
├── .gitignore                 # Git ignore rules
├── imported_resources.md      # Documentation of imported resources
├── README.md                  # Project README with build instructions
├── book.toml                  # mdBook configuration
├── .terraform/                # Terraform provider binaries (generated)
├── .terraform.lock.hcl        # Provider version lock file
├── terraform.tfstate          # State file (not in version control)
├── terraform.tfstate.backup   # State backup (not in version control)
└── modules/
    └── snowflake/
        ├── roles/
        │   ├── main.tf        # Role resource definitions
        │   ├── variables.tf   # Role module variables
        │   └── outputs.tf     # Role module outputs
        ├── users/
        │   ├── main.tf        # User resource definitions
        │   ├── variables.tf   # User module variables
        │   └── outputs.tf     # User module outputs
        ├── warehouse/
        │   ├── main.tf        # Warehouse resource definitions
        │   ├── variables.tf   # Warehouse module variables
        │   └── outputs.tf     # Warehouse module outputs
        ├── database/
        │   ├── main.tf        # Database resource definitions and grants
        │   ├── variables.tf   # Database module variables
        │   └── outputs.tf     # Database module outputs
        ├── schema/
        │   ├── main.tf        # Schema resource definitions and grants
        │   ├── variables.tf   # Schema module variables
        │   └── outputs.tf     # Schema module outputs
        ├── tables/
        │   ├── main.tf        # Table resource definitions and grants
        │   ├── variables.tf   # Tables module variables
        │   └── outputs.tf     # Tables module outputs
        └── views/
            ├── main.tf        # View resource definitions and grants
            ├── variables.tf   # Views module variables
            └── outputs.tf     # Views module outputs

File Descriptions

Root Level Files

main.tf

The primary configuration file containing:

  • Module declarations for all resources (roles, users, warehouses, databases, schemas, tables, views)
  • All resources are managed through modules - no direct resource definitions
  • Each module handles its own privilege grants automatically
  • Map-based configuration for each module (role_map, user_map, warehouse_map, database_map, schema_map, table_map, view_map)

provider.tf

Defines the Terraform providers:

  • Snowflake provider configuration
  • Provider aliases for different roles (SECURITYADMIN, SYSADMIN)
  • Required provider versions

variables.tf

Defines all input variables:

  • Snowflake connection parameters
  • Default values
  • Variable types and descriptions

outputs.tf

Defines outputs from the root module:

  • Module outputs
  • Resource references
  • Sensitive output flags

terraform.tfvars

Contains actual variable values:

  • Not committed to version control (in .gitignore)
  • Contains sensitive credentials (snowflake_password, etc.)
  • Environment-specific values
  • Required variables: snowflake_organization, snowflake_account, snowflake_username, snowflake_password, snowflake_role, snowflake_warehouse

imported_resources.md

Documents resources that have been imported into Terraform management:

  • Tracks imported resources for reference
  • Contains import commands for future use
  • Helps maintain awareness of manually imported resources

README.md

Project README file:

  • Instructions for building mdBook documentation
  • Quick start guide
  • Project overview

book.toml

mdBook configuration file:

  • Defines book title, description, and metadata
  • Configures HTML output settings
  • Sets search and theme preferences

Module Structure

The project uses a fully modular architecture with seven main modules:

modules/snowflake/roles/

Manages Snowflake account roles:

  • Creates roles with hierarchical relationships
  • Grants roles to parent roles
  • Outputs role information

modules/snowflake/users/

Manages Snowflake users:

  • Supports person users, service users, and legacy service users
  • Assigns roles to users
  • Configures user defaults (warehouse, role)

modules/snowflake/warehouse/

Manages Snowflake warehouses:

  • Creates warehouses with configurable properties
  • Grants warehouse privileges to roles automatically
  • Supports multiple warehouse configurations

modules/snowflake/database/

Manages Snowflake databases:

  • Creates databases with retention policies
  • Grants database privileges to roles automatically
  • Supports multiple databases with different configurations

modules/snowflake/schema/

Manages Snowflake schemas:

  • Creates schemas within databases
  • Supports managed access and transient schemas
  • Grants schema privileges to roles automatically

modules/snowflake/tables/

Manages Snowflake tables:

  • Creates tables with column definitions
  • Supports various data types and constraints
  • Grants table privileges to roles automatically

modules/snowflake/views/

Manages Snowflake views:

  • Creates views with SQL statements
  • Supports table references in SQL
  • Grants view privileges to roles automatically

Design Principles

1. Modularity

Resources are organized into reusable modules:

  • Reusability: Modules can be used across different configurations
  • Maintainability: Changes to module logic are centralized
  • Testability: Modules can be tested independently

2. Separation of Concerns

Different resource types are managed by different modules:

  • Roles Module: Handles role creation and hierarchy
  • Users Module: Handles user creation and role assignment
  • Warehouse Module: Handles warehouse creation and privileges
  • Database Module: Handles database creation and privileges
  • Schema Module: Handles schema creation and privileges
  • Tables Module: Handles table creation and privileges
  • Views Module: Handles view creation and privileges

3. Provider Aliases

Different provider aliases for different operations:

  • SECURITYADMIN: For role and user management
  • SYSADMIN: For warehouse, database, and object management

4. Map-Based Configuration

Resources are defined using maps for flexibility:

  • Easy to add/remove resources
  • Consistent structure
  • Easy to iterate over

Resource Dependencies

The following dependency order is enforced:

Roles
  ↓
Warehouses (references roles for grants)
  ↓
Users (references roles and warehouses)
  ↓
Databases (references roles for grants)
  ↓
Schemas (references databases and roles for grants)
  ↓
Tables (references databases, schemas, and roles for grants)
  ↓
Views (references databases, schemas, tables, and roles for grants)

Note: All grants are handled automatically within each module, so there are no separate grant resources in the root configuration. Each module manages its own privilege grants internally.

State Management

Local State (Default)

  • Stored in terraform.tfstate
  • Backup in terraform.tfstate.backup
  • Not committed to version control

Remote State (Recommended for Teams)

Consider using:

  • Terraform Cloud: Free tier available
  • AWS S3: With DynamoDB for locking
  • Azure Storage: With state locking
  • Google Cloud Storage: With state locking

Version Control

Committed Files

  • *.tf files (configuration)
  • .gitignore
  • .terraform.lock.hcl (provider versions)

Ignored Files

  • *.tfvars (contains secrets)
  • terraform.tfstate* (state files)
  • .terraform/ (provider binaries)

Extension Points

Adding New Modules

  1. Create directory: modules/snowflake/<module-name>/
  2. Add main.tf, variables.tf, outputs.tf
  3. Reference in root main.tf

Adding New Resources

  1. Add resource definition to appropriate file
  2. Add variables if needed
  3. Add outputs if needed
  4. Update documentation

Environment-Specific Configurations

Use separate .tfvars files:

  • terraform.tfvars.dev
  • terraform.tfvars.staging
  • terraform.tfvars.prod

Apply with: terraform apply -var-file="terraform.tfvars.dev"

Best Practices

  1. Keep Modules Focused: Each module should have a single responsibility
  2. Use Variables: Don’t hardcode values
  3. Document Dependencies: Use depends_on explicitly when needed
  4. Version Control: Commit all .tf files, never commit .tfvars or state
  5. Remote State: Use remote state for team collaboration
  6. Code Review: Review all changes before applying
